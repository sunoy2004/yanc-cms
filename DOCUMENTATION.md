## YANC CMS – Full Project Documentation

This document gives a **complete overview of the YANC CMS project** – architecture, project structure, major modules, and file‑by‑file responsibilities – so that a new developer can start working and extending it without previous context.

---

## 1. High‑Level Overview

**YANC CMS** is a small, self‑hosted content management system for the YANC website. It consists of:

- **Frontend CMS app**
  - Tech: **Vite + React + TypeScript**, **React Router**, **shadcn‑ui**, **Tailwind CSS**.
  - Location: root `src/`, `index.html`, `vite.config.*`, etc.
  - Purpose: Admin UI for managing hero content, programs, events (upcoming & past), event gallery, team/founders, testimonials, about content, media library, mentor talks, and dashboard stats.

- **Backend API**
  - Tech: **NestJS** (modular monolith), **Supabase (Postgres + Storage)**.
  - Location: `apps/cms-api/`.
  - Purpose: JSON API for the CMS frontend and public site; handles auth, data access, publishing/drafts, upload URLs, and health/stats.

- **Database layer**
  - Tech: **Postgres on Supabase**.
  - Migrations: Single master SQL file `master-migrations.sql` (root), to be run in Supabase SQL editor.
  - Organizes content into tables like `hero_content`, `program_content`, `event_content`, `event_gallery_items`, `media`, `team_members`, `founders`, `testimonials`, `about_us`, `mentor_talks`, etc.

Supporting pieces include Cloud Build config (`cloudbuild.yaml`), Nginx config (`nginx.conf`), lints/tests (`eslint`, `vitest`), and a design/requirements folder (`cms-design/`).

## 3. How to Run the Project

### 3.1. Prerequisites

- Node.js (LTS) + npm.
- Supabase project (Postgres + Storage) and corresponding service role key.

### 3.2. Database Setup

1. Open Supabase SQL editor for your project.
2. Copy the contents of **`master-migrations.sql`**.
3. Run the script once on an empty/new database.
4. Configure environment variables for the backend:
   - `SUPABASE_URL`
   - `SUPABASE_SERVICE_KEY`
   - Any other secrets referenced in `apps/cms-api/src/supabase/supabase.service.ts` (see that file for details).

### 3.3. Backend API (NestJS) – `apps/cms-api`

- **Install backend deps** (from repo root or inside `apps/cms-api`):

```bash
cd apps/cms-api
npm install
```

- **Run dev server**:

```bash
npm run start:dev
```

This starts NestJS on the port configured in `main.ts` (typically `http://localhost:3001/api`).

### 3.4. Frontend CMS (Vite React) – root

From repo root:

```bash
npm install
npm run dev
```

Open the Vite dev URL (usually `http://localhost:5173`). The frontend uses `VITE_CMS_BASE_URL` (see `.env.*`) to talk to the API at `/api/...` routes.

## 4. Backend API – Detailed Structure (`apps/cms-api`)

### 4.1. Key Backend Files

- **`apps/cms-api/src/main.ts`**
  - Bootstraps a NestJS app with:
    - CORS enabled (for local dev).
    - Validation pipe (DTO validation).
    - Global prefix `api` (all routes are under `/api/...`).
    - Logging of startup info.

- **`apps/cms-api/src/app.module.ts`**
  - Root module; imports:
    - `ConfigModule` – environment variables.
    - `SupabaseModule` – wraps Supabase client.
    - Feature modules: `AuthModule`, `StatsModule`, `MediaModule`, `HeroModule`, `ProgramsModule`, `EventsModule`, `TeamModule`, `FoundersModule`, `TestimonialsModule`, `AboutModule`, `EventGalleryItemsModule`, `MentorTalksModule`, `ActivityModule`, `DeployModule`.
  - Declares controllers:
    - `AppController` – root routes / health.
    - `HealthController` – database health checks.
  - Provider:
    - `DatabaseHealthService` – verifies DB connectivity.

- **`apps/cms-api/src/supabase/supabase.module.ts` / `supabase.service.ts`**
  - Centralizes **Supabase client** creation with credentials from env vars.
  - Exposes `getClient()` that other services call for DB operations.

- **`apps/cms-api/src/common/health.controller.ts`**
  - Exposes routes like `/api/health` or `/api/database-health` for monitoring.

- **`apps/cms-api/src/common/decorators/public.decorator.ts`**
  - NestJS decorator `@Public()` sets metadata so `JwtAuthGuard` skips auth for those routes.

- **`apps/cms-api/src/common/database-health.service.ts`**
  - Utility to run minimal queries against Supabase to validate DB connectivity.

### 4.2. Auth Module – `apps/cms-api/src/modules/auth/*`

- **`auth.module.ts`**
  - Declares services/guards/controllers for authentication.
- **`auth.controller.ts`**
  - Endpoints like login, password reset, etc. (exact routes can be confirmed in this file).
- **`auth.service.ts`**
  - Uses `bcryptjs` and Supabase `users` table for:
    - Verifying credentials.
    - Issuing JWTs.

- **`guards/jwt-auth.guard.ts`**
  - Protects endpoints by validating JWT and attaching user info to the request.

### 4.3. Hero Module – `apps/cms-api/src/modules/hero/*`

- **`hero.controller.ts`**
  - Routes to list, create, update, publish hero blocks.
- **`hero.service.ts`**
  - Uses `hero_content` and `hero_media_item` tables.
  - Maps between DB structure and frontend DTOs.
  - Handles publish/draft via `is_active` column.

- **DTOs** (`apps/cms-api/src/dtos/hero*.ts`)
  - `HeroDto` and `HeroUpdateDto` shape the data for creation and update from CMS.

### 4.4. Programs Module – `apps/cms-api/src/modules/programs/*`

- **`programs.controller.ts`**
  - Endpoints:
    - `GET /api/programs` – list for CMS (with drafts).
    - `GET /api/programs/public` – published only for frontend site.
    - `POST /api/programs` – create.
    - `PUT /api/programs/:id` – update.
    - `DELETE /api/programs/:id` – delete.
    - `PATCH /api/programs/:id/publish` – toggle publish.

- **`programs.service.ts`**
  - CRUD against `program_content` table using Supabase.
  - Uses `is_active` as published flag.
  - Populates `isPublished` and media info for CMS UI.

- **DTOs**: `apps/cms-api/src/dtos/program-update.dto.ts`, `content.dto.ts`.

### 4.5. Events Module – `apps/cms-api/src/modules/events/*`

- **`events.controller.ts`**
  - `GET /api/events/public` – full list (CMS & public).
  - `GET /api/events/upcoming` – upcoming events; also triggers auto‑drafting of past upcoming events.
  - `GET /api/events/past` – past events.
  - `GET /api/events/highlights` – events with highlights.
  - `GET /api/events/by-year/:year` – filter by year.
  - `POST /api/events` – create event (decorated with `@Public()` for ease of use).
  - `PUT /api/events/:id` – update.
  - `DELETE /api/events/:id` – delete.
  - `PATCH /api/events/:id/publish` – toggle `published`/`is_active`.

- **`events.service.ts`**
  - `getEvents()`
    - Queries `event_content` with joins to:
      - `event_highlights` (content list).
      - `event_gallery` / media via `event_gallery` and `media` tables.
    - Adapts DB data to frontend shape:
      - `isPublished` based on `is_active` plus additional logic.
      - `isPast`, `isUpcoming`, `year`, `month` computed from `event_date`.
    - Calls `draftPastUpcomingEvents()` to auto‑draft past upcoming events when fetching.
  - `createEvent(dto)`
    - Inserts into `event_content` (with `registration_url`, `category` = `upcoming` or `past`).
    - Inserts associated gallery entries into `event_gallery` and highlights into `event_highlights`.
  - `updateEvent(id, dto)`
    - Updates event fields, gallery, highlights.
    - Handles `registration_url` presence/absence across environments.
  - `deleteEvent(id)`
    - Deletes from `event_content` (cascade removes gallery/highlights).
  - `draftPastUpcomingEvents()`
    - `UPDATE event_content SET is_active = false` where `category = 'upcoming'`, `is_active = true`, and `event_date < now()`.
    - Used to automatically mark upcoming events whose date has passed as “Draft” in CMS.

- **DTOs**: `event.dto.ts`, `event-update.dto.ts`.

### 4.6. Event Gallery Items Module – `apps/cms-api/src/modules/event-gallery-items/*`

- **`event-gallery-items.controller.ts`**
  - Endpoints for managing a separate **Event Gallery** section that is decoupled from `event_content`:
    - List public gallery items.
    - CRUD for CMS.
    - Publish toggle via `/event-gallery-items/:id/publish`.

- **`event-gallery-items.service.ts`**
  - Uses `event_gallery_items` and `event_gallery_item_media` tables.
  - Maps post‑processed Supabase data into gallery items with associated media.

### 4.7. Team Module – `apps/cms-api/src/modules/team/*`

- **`team.controller.ts`**
  - Endpoints:
    - `GET /team` – CMS list.
    - `GET /team/public` – public site list, with grouping by sections.
    - `PATCH /team/:id/publish` – toggle publish.

- **`team.service.ts`**
  - Works with `team_members`, `team_member_media_item` and `social_links` tables.
  - Returns CMS‑friendly objects including:
    - `isPublished` (from `is_active`).
    - `section` mapping to UI sections (executive, founders, advisory, mentors).

### 4.8. Founders Module – `apps/cms-api/src/modules/founders/*`

- Similar pattern to Team:
  - CRUD for `founders` table.
  - `togglePublish` endpoint updates `is_active`.
  - DTOs: `founder.dto.ts`, `founder-update.dto.ts`.

### 4.9. Testimonials Module – `apps/cms-api/src/modules/testimonials/*`

- CRUD for `testimonials` plus publish toggle.
- DTOs: `testimonial.dto.ts`, `testimonial-update.dto.ts`.

### 4.10. About Module – `apps/cms-api/src/modules/about/*`

- Manages `about_us` content blocks including hero text and sections.
- Supports publish toggling such that only one about block is active at a time (`is_active = true` logic).

### 4.11. Mentor Talks Module – `apps/cms-api/src/modules/mentor-talks/*`

- **`mentor-talks.controller.ts`**
  - `GET /mentor-talks` – admin list.
  - `GET /mentor-talks/public` – public published list.
  - `POST /mentor-talks` – create.
  - `PUT /mentor-talks/:id` – update.
  - `PATCH /mentor-talks/:id/publish` – toggle publish.

- **`mentor-talks.service.ts`**
  - Works with `mentor_talks` table (strict schema from `master-migrations.sql`).
  - Exposes derived fields: `isPublished`, `talk_date`, etc.

### 4.12. Media Module – `apps/cms-api/src/modules/media/*`

- Central layer handling Media table and generating public URLs via Supabase Storage paths.
- Used by other modules (hero, events, gallery, team) via relationships like `hero_media_item`, `event_gallery_item_media`, etc.

### 4.13. Stats Module – `apps/cms-api/src/modules/stats/*`

- **`stats.service.ts`**
  - Computes dashboard statistics for the CMS:
    - Total and published counts for events, team, programs, testimonials, etc.
  - Uses Supabase `count` queries with `is_active` filters.

### 4.14. Activity & Deploy Modules

- **Activity module** – records admin activities (e.g. content changes) into `activity` table (see its service for exact shape).
- **Deploy module** – convenience endpoints for triggering rebuilds/deploy hooks in hosting environment (e.g. calling Cloud Build or Vercel hooks).

## 5. Database Schema – `master-migrations.sql`

The **single source of truth** for the DB schema:

- **Core tables**
  - `media`, `hero_content`, `program_content`, `event_content`, `event_gallery`, `mentor_talks`, `team_members`, `founders`, `testimonials`, `about_us`, `contact_info`, `core_values`, `faqs`, `careers`.

- **Extended tables**
  - `applications`, `newsletter_subscriptions`, `gallery`, `gallery_items`, `social_links`.
  - Media relation tables: `hero_media_item`, `program_media_item`, `event_media_item`, `event_gallery_media_item`, `mentor_talk_media_item`, `team_member_media_item`, `founder_media_item`, `testimonial_media_item`, `about_us_media_item`, `gallery_media_item`.

- **Events & gallery isolation**
  - `event_content` has `type` (`upcoming` | `past`) and `category`, `event_date`, `is_active`, `registration_url`.
  - `event_gallery_items` and `event_gallery_item_media` decouple gallery content from event rows.
  - `event_highlights` table stores per‑event highlight bullets for display.

- **Team sections**
  - `team_members.section` with constraint (`executive_management`, `cohort_founders`, `advisory_board`, `global_mentors`).

- **Users & auth**
  - `users` table with `auth_id`, `username`, `role` (`admin` or `editor`), `reset_token`, `reset_token_expires_at`, `created_by`.

- **RLS & policies**
  - Every content table has RLS enabled with a **“service role can access X”** policy for the backend.
  - Some public tables (e.g. event gallery items, mentor talks) have public `SELECT` policies for published rows.

- **Triggers**
  - Generic `update_updated_at_column()` trigger function attached to almost every table to auto‑update `updated_at` on modification.

> Any schema change should be applied by editing `master-migrations.sql` and re‑running it on a fresh DB, or by authoring a new one‑off migration and then merging it back into the master script.

## 7. Developer Workflow & Conventions

### 7.1. Coding Style

- **TypeScript everywhere** – avoid `any` when possible.
- **DTOs + services** pattern in backend:
  - Each module has DTOs in `apps/cms-api/src/dtos` and a corresponding `*.service.ts`/`*.controller.ts` pair.
  - API returns objects shaped for CMS convenience (e.g. `isPublished`) not raw DB rows.

- **React + hooks** in frontend:
  - Pages are React function components using `useState`, `useEffect`, `react-hook-form` and `React Query` (where added) to fetch data.

### 7.2. Adding a New CMS Section (Example)

1. **Database**
   - Add new table + RLS/policies to `master-migrations.sql`.
   - Run migration on Supabase.

2. **Backend**
   - Create new module under `apps/cms-api/src/modules/<feature>/` with:
     - `*.module.ts`, `*.controller.ts`, `*.service.ts`.
   - Add DTOs under `apps/cms-api/src/dtos/<feature>.dto.ts`.
   - Register module in `AppModule` imports.

3. **Frontend**
   - Add a service wrapper in `src/services/<feature>.service.ts` that calls the new `/api/<feature>` endpoints.
   - Create a page under `src/pages/<Feature>.tsx` using `DataTable`, `PageHeader`, and forms.
   - Wire into routes in `src/App.tsx` and nav link in `CMSSidebar`.

4. **Testing & docs**
   - Add basic `vitest` tests for business logic (optional but recommended).
   - Update this `DOCUMENTATION.md` and/or `cms-design/api-routes.md` with new routes and behaviors.
